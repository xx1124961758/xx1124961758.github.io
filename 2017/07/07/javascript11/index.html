每一个开发者都在努力写出具有可维护性，可读性，可复用的代码。随着应用规模的变大，代码的结构化变得更加重要。设计模式 （ 在一种在特定场景下，为普遍的问题提供一个组织结构 ） 是解决这一挑战的关键。

JavaScript web 开发者在创建应用的时候，经常会用到设计模式 （ 即使他们没有意识到 ）。

虽然在某些情况下使用到的设计模式很多样化，但是 JavaScript 开发者通常比其他人更倾向使用一些设计模式。

在这篇文章中，我想讨论下一些普遍的设计模式，以提高你们的编程能力和深入 JavaScript 内部。

将要讨论的设计模式有：

  • 模块（Module）
  • 原型（Prototype）
  • 观察者（Observer）
  • 单例（Singleton）

每一种设计模式都包含很多属性，不过，我会着重强调下面几个关键点：

  1. 上下文（Context）: 在什么情况下使用或者在哪里使用？
  2. 问题（Problem）: 我们会尝试解决什么问题？
  3. 解决方法（Solution）: 用这个设计模式解决我们提出的问题怎么样？
  4. 实现（Implementation）: 实现起来是什么样的？

”模块“设计模式

JavaScript 模块是普遍用来保持特定的代码片段独立于其他组件的设计模式。这提供了松散耦合，结构良好的代码。

原型设计模式

任何一个 JavaScript 开发者看到过这个 prototype 关键字。被原型继承或者在他们的代码中实现继承所困扰过。原型设计模式依赖于 JavaScript 的原型继承。原型模式大多数被用来高性能地创建对象。

观察者模式

当程序的一部分变化了，其他部分也需要变化。这种情况，在开发中很常见。在 AngularJS 中，如果 $scope 更新了，就会触发一个事件，以通知其他的组件。观察者模式就是：如果一个对象发生了变化，那就把变化的信息广播（broadcasts）给其他依赖的对象。

另一个基本的例子是：MVC 架构。当 model 变化后，view 视图也跟着变化。一个好处就是解耦 View 和 Model，减少他们的相关性。

正如图片描述的那样，相关的对象有：subjext，observer 和 concrete。subject 对象包含了 concrete 对象为每一次变化都通知的观察器。observer 对象是一个抽象类，允许 concrete 观察器实现通知的方法。

单例模式

单例模式只允许一个实例对象，但可以有同一个对象的多个实例存在。单例模式限制创建多个对象，当第一个对象被创建完毕之后，它将会返回同一个对象。

对之前没有用过单例模式的同学找个例子不好找啊。一个例子就是：使用公司打印机。如果一个公司有10个人，都共用一个打印机，10台电脑共享一台打印机。通过共享一个打印机，他们共享同样的资源。

在多线程的应用中，有超过一个线程试着访问资源。单例模式被竞争条件所影响。如果第一次没有初始化，两个线程会创建两个对象，而不是返回和初始化，这个是单例的 目的，因此当在多个线程的应用中应用单例，同步处理是私有的。

结论

设计模式经常被用于大型的应用开发中，通过理解一种模式在某种场景下比另一种好，然后实践。

在创建一个应用之前，你应该考虑下他们的角色，以及如何交互。复习完 Module，Prototype，Observer和 Singleton 设计模式，你应该能够辨别他们，并运用到实际开发中。
